# Plan: User-Editable Tile Title Field (Android + Web)

## Goal

Add a user-editable **title** field to every tile, on both the Android and web versions of HubitatDashboard. The title is set in edit mode. If the title is blank, the device name (hub label) is displayed — which is the current behavior. If the title is filled in, it replaces the device name in the tile's visible header.

## Current Context / Assumptions

- **TileConfig** (Android `data class TileConfig` and Web `interface TileConfig`) both carry a `label` field that is always derived from `DeviceState.label` (the hub device name). It's never user-editable.
- Each tile type already renders its label: Android via `TileShell(title = tile.label)`, Web via inline `<p className="text-xs ...">{label}</p>`.
- Edit mode exists on both platforms:
  - **Web**: `EditOverlay` component in `GroupPage.tsx` shows device name + Remove/Add-to/Type buttons.
  - **Android**: `GroupScreen.kt` shows Close (X) and Tune icons over each tile.
- Persistence is per-client: Android uses SharedPreferences (JSON/Gson), Web uses Zustand persist (localStorage). Sync between them works via `GroupExportData` JSON files (push/pull through hub file manager).
- The existing override pattern (mirrored on both sides) is `tileTypeOverrides: groupId → deviceId → TileType`. We will add `tileTitleOverrides: groupId → deviceId → string` following the same pattern.

## Proposed Approach

Add a `tileTitleOverrides` map (groupId → deviceId/syntheticId → title string) to both platforms. In the tile resolution/display pipeline, use the override title if present; otherwise fall back to the device name. In edit mode, add a small text input so the user can set or clear the title.

The title is **per-group** (same device can have different titles in different groups), matching the existing tileTypeOverrides semantics.

## Step-by-Step Plan

### Phase 1: Data Model + Persistence

#### 1.1 Web — groupStore.ts

- Add `tileTitleOverrides: Record<string, Record<string, string>>` to the `GroupStore` interface.
- Initialize as `{}` in the store defaults.
- Add action: `setTileTitle(groupId: string, deviceId: string, title: string) => void`
  - Sets `tileTitleOverrides[groupId][deviceId] = title` (or deletes the key if title is empty)
  - Persisted automatically via Zustand persist.
- Update `GroupExportPayload` interface to include `tileTitleOverrides`.
- Update `importState` to accept and store `tileTitleOverrides`.
- Update the `merge` function in persist config to restore `tileTitleOverrides` from localStorage.

**File**: `frontend/src/store/groupStore.ts`

#### 1.2 Web — types.ts

- Add optional `title?: string` to `TileConfig` interface (the display-resolved field, not stored — computed from overrides).

**File**: `frontend/src/types.ts`

#### 1.3 Android — GroupRepository.kt

- Add `KEY_TILE_TITLE_OVERRIDES = "tile_title_overrides"` companion constant.
- Add `_tileTitleOverrides: MutableStateFlow<Map<String, Map<String, String>>>` initialized from prefs.
- Add `setTileTitle(groupId, deviceId, title)` method — stores or removes entry.
- Add `tileTitleOverridesRaw` accessor for export.
- Update `saveAll()` to persist the new field.
- Update `replaceAll()` to accept and restore `tileTitleOverrides`.
- Add `_tileTitleOverrides` to the `resolvedGroupsFlow` combine chain.
- In `resolveGroupsNow()`, when building `TileConfig`, set `label` from the override if present. Also pass the raw device name separately (see 1.5).

**File**: `android/app/src/main/.../data/repository/GroupRepository.kt`

#### 1.4 Android — TileConfig.kt

- Add `title: String? = null` field. This holds the user-editable title (null or blank = use label as-is).

**File**: `android/app/src/main/.../data/model/TileConfig.kt`

#### 1.5 Android — GroupRepository resolution

- In `resolveGroupsNow()`, after building tiles, apply title overrides:
  ```
  val titleMap = tileTitleOverrides[groupId] ?: emptyMap()
  tiles = tiles.map { tile ->
      val overrideTitle = tile.deviceId?.let { titleMap[it] }
      if (!overrideTitle.isNullOrBlank()) tile.copy(title = overrideTitle) else tile
  }
  ```

#### 1.6 Android — GroupEditViewModel.kt

- Add `setTileTitle(groupId: String, deviceId: String, title: String)` method that delegates to `groupRepository.setTileTitle(...)`.

**File**: `android/app/src/main/.../viewmodel/GroupEditViewModel.kt`

### Phase 2: Display — Use Title When Present

#### 2.1 Web — GroupPage.tsx (tile resolution)

- In `TileWrapper` or the tile-building section, compute the effective title:
  ```ts
  const titleOverride = tileTitleOverrides[groupId]?.[effectiveTile.deviceId ?? '']
  const displayLabel = (titleOverride && titleOverride.trim()) || effectiveTile.label
  ```
- Pass `displayLabel` as the `label` prop to each tile component (already done via `base.label` in `renderTile`).

**File**: `frontend/src/components/GroupPage.tsx`

#### 2.2 Web — All Tile Components (no changes needed)

- All web tile components already receive `label` as a prop and render it. Since we override the label before passing it in, individual tile files don't need modification.

#### 2.3 Android — All Tile Composables (no changes needed)

- All Android tiles already use `tile.label` via `TileShell(title = tile.label, ...)`.
- **However**: The `label` field still holds the device name. The display logic should use `tile.title` if non-blank, else `tile.label`. This can be done in one place:
  - Option A: In `TileShell`, change signature to accept both `title` and `fallbackTitle`, display `title.ifBlank { fallbackTitle }`.
  - Option B: In `TileCard`, compute the effective display name once and pass it down.
  - **Recommended**: Add a helper extension/property: `TileConfig.displayTitle: String get() = if (title.isNullOrBlank()) label else title`
  - Then update each tile's `TileShell(title = tile.displayTitle, ...)` call — or better, update TileShell itself.

**Files** (16 tile files + common/TileShell.kt):
- `common/TileShell.kt` — add `TileConfig.displayTitle` extension or modify TileShell params
- All `ui/tiles/*Tile.kt` files — change `tile.label` to `tile.displayTitle`

### Phase 3: Edit Mode UI — Title Input

- On blur or Enter key, call `setTileTitle(groupId, deviceId, titleDraft)`.
- Placeholder text: device name (so user sees what will show if they clear the title).
- The `EditOverlay` already gets `tile`, `groupId` — just needs access to `setTileTitle` and `tileTitleOverrides` from the store.

**File**: `frontend/src/components/GroupPage.tsx`

#### 3.2 Android — GroupScreen.kt (edit mode title editing)

- In the edit-mode overlay section (lines ~286–330), add a clickable title area:
  - When tapped, show a `TextField` dialog (or inline `OutlinedTextField`) to edit the title.
  - Display current title (or device name as placeholder).
  - On confirm, call `groupEditViewModel.setTileTitle(groupId, deviceId, newTitle)`.
- The edit surface could be a `BasicAlertDialog` with an `OutlinedTextField` — standard Material 3 pattern.

**File**: `android/app/src/main/.../ui/group/GroupScreen.kt`

## Tests / Validation

1. **Functional test (Web)**: Enter edit mode → click a tile's title area → type a custom title → exit edit mode → confirm tile shows custom title. Clear the title → confirm device name is shown again.
2. **Functional test (Android)**: Enter edit mode → tap title area → dialog appears → enter title → confirm → tile shows custom title. Clear → device name appears.
3. **Persistence test**: Set a title, refresh the web page / kill and restart Android app → title persists.
4. **Cross-platform sync**: Export config from web → import into Android (or push/pull via hub) → titles survive the round-trip.
5. **Build verification**: `bash gradlew assembleDebug` in `android/` compiles without errors. Web frontend builds with `npm run build` in `frontend/`.

## Risks, Tradeoffs, Open Questions

1. **Tile key for synthetic tiles**: Special tiles use synthetic IDs like `__hsm__`, `__sunrise__`, etc. The title overrides should be keyed by these synthetic IDs so users can rename them too. This is already handled since the edit overlay has access to the `deviceId` or synthetic ID.

2. **Multi-device tiles**: These already have their own `label` in `MultiTileConfig`. Should titles also be editable for these? The user said "all of them" — so yes. The multi-tile title might be best handled via the `MultiTileConfig.label` field rather than the new override system, since multi-tiles have a dedicated config store.

3. **Title overflow**: Tile titles are already `truncate` (single line with ellipsis). Long custom titles will be truncated in display, which is acceptable.

4. **Backward compatibility**: Old export JSON files won't have `tileTitleOverrides`. Default to empty map on import — no migration needed.

5. **Android build**: The Android app needs the `TileConfig.displayTitle` property to be accessible. Using a Kotlin extension property on the data class is cleanest and doesn't require changing each tile's signature.

6. **Open question**: If the user sets a title on web, does it sync to Android automatically or only via explicit push/pull? Currently tileTypeOverrides only sync via push/pull — same behavior applies here. No real-time sync.
