# Plan: GPS Map Page (Google Sheet → Google Maps)

## Goal
Add a new `/gps-map` page to the HubitatDashboard that reads GPS data (timestamp, lat, long) from a Google Sheet (`GPS_Log` tab, Sheet ID `1EGNmf9XvinmTE5EGZUOBjaoeU1HIvBLHlz33TN0KfcU`) and displays it as a line track on Google Maps with date range filtering. This must NOT disturb the existing dashboard functionality.

## Current Context & Assumptions

- **Frontend** (`frontend/`): React 19, TypeScript, Vite 8, Tailwind CSS 4, react-router-dom 7, Zustand stores

- **Monorepo root**: npm workspaces (`concurrently` for dev), no direct app code at root
- **API pattern**: Frontend proxies `/api/*` → backend at port 3001 (configured in `vite.config.ts`)
- **Config pattern**: Backend reads `backend/config.json` (see `backend/src/config.ts`); some values overridden by env vars (e.g., `DATABASE_URL`)
- **Navigation**: Sidebar with group-based pages; routes defined in `frontend/src/App.tsx`
- **Current routes**: `/` → `/group/environment`, `/group/:groupId` → GroupPage

## Configuration Strategy

### Backend config (extends `backend/config.json`)
Add a `gpsMap` section:
```json
{
  "hubIP": "...",
  ...existing fields...,
  "gpsMap": {
    "googleSheetId": "1EGNmf9XvinmTE5EGZUOBjaoeU1HIvBLHlz33TN0KfcU",
    "sheetRange": "GPS_Log!A:C",
    "serviceAccountKeyPath": "/path/to/service-account-key.json",
    "googleMapsApiKey": "YOUR_KEY"
  }
}
```
- `serviceAccountKeyPath`: path to the JSON key file (user responsibility to create/store securely, outside repo)
- `googleMapsApiKey`: client-side key for Maps JS API (will be served to frontend via a config endpoint or VITE_ env var)
- Both are optional to avoid breaking existing dashboard startup if not configured

### Frontend Google Maps API Key
- Pass via Vite env: `VITE_GOOGLE_MAPS_API_KEY` in `.env.local`
- Or fetch from a `/api/gps-map/config` endpoint that returns just the key
- `.env.local` approach is simpler and consistent with Vite conventions

### User Responsibilities (sensitive data)
- Create Google Cloud service account, download key JSON, place outside repo
- Set Maps API key in frontend `.env.local`
- Share the target Google Sheet with the service account email

## Proposed Approach

### Data Access Strategy (two options, recommend Option A)

**Option A: Google Sheets API directly** (more robust, server-side only)
- Backend uses `googleapis` package with service account credentials
- Reads sheet data via Sheets API v4
- Pro: Full control, filtering, no dependency on external script

**Option B: Fetch from Apps Script web app URL** (simpler, no service account needed for this step)
- Backend just HTTP-GETs the Apps Script URL and parses JSON response
- Pro: Zero auth setup if the web app is publicly accessible
- Con: Depends on the script staying deployed and its output format
- **Will try Option B first** to validate data format, then implement A as primary

## Risks, Tradeoffs & Open Questions

- **Risk: Service account setup required.** The user must create a GCP service account, grant it Sheets Reader role, and share the sheet with the service account email. Without this, the backend can't read data.
  - Mitigation: Apps Script web app URL as fallback (Option B) — test it first with `curl`.
- **Risk: Timestamp parsing.** Unknown format of the `timestamp` column in Google Sheets. If it's a raw date string vs Excel serial number, parsing logic needs adjustment.
  - Mitigation: Fetch a sample row first, detect format, write appropriate parser.
- **Risk: Large dataset performance.** If the sheet has thousands of GPS points, the map could be slow.
  - Mitigation: Server-side date filtering reduces payload; consider downsampling for display if needed.
- **Tradeoff: Separate page vs integrated.** Putting GPS map as a standalone route (not a group tile) keeps it completely isolated from dashboard logic, per user's request to "not disturb."
- **Open Question:** What timestamp format does the Google Sheet use? (Will be discovered during implementation.)
- **Open Question:** Should the service account key path be in `config.json` or env var? (Recommend config.json to match existing pattern.)
